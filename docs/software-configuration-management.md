# Software Configuration Management

## Purpose

This document defines the Software Configuration Management strategy for BioTrack Web Application. It describes how the team manages source code, development environments, deployment configuration, dependencies, environment variables and frontend/backend runtime compatibility throughout the project lifecycle.

## Source Code Management

| Tool | Purpose | Official URL |
| --- | --- | --- |
| Git | Distributed version control for source code, configuration and documentation changes. | https://git-scm.com/ |
| GitHub | Remote repository hosting, collaboration, code review and project traceability. | https://github.com/ |
| GitHub Pages | Static hosting for the production frontend build generated by Vite. | https://pages.github.com/ |

### Repository Conventions

- Main source code lives under `src/`.
- Business capabilities are organized by bounded context.
- Shared technical infrastructure lives under `src/shared`.
- Runtime environment configuration is centralized in `src/config/env.js`.
- API access is centralized in `src/shared/infrastructure/api.service.js`.
- Environment variables must use the `VITE_` prefix to be exposed by Vite.
- Local-only secrets must not be committed. Current variables are public configuration values only.

## Environment Files

BioTrack uses Vite environment files to separate local development from production deployment.

```text
.env
.env.development
.env.production
.env.example
src/config/env.js
src/shared/infrastructure/api.service.js
```

### Variables

| Variable | Purpose | Development Example | Production Example |
| --- | --- | --- | --- |
| `VITE_APP_ENV` | Human-readable application environment label. | `development` | `production` |
| `VITE_APP_BASE_PATH` | Public base path used by Vite and GitHub Pages. | `/biotrack-frontend/` | `/biotrack-frontend/` |
| `VITE_API_BASE_URL` | Backend root URL used by Axios. | `http://localhost:3001` | `https://TU_BACKEND_PUBLICO.com` |
| `VITE_API_TIMEOUT` | HTTP request timeout in milliseconds. | `8000` | `10000` |
| `VITE_ENABLE_API_DEBUG` | Enables environment and API error logs for debugging. | `true` | `false` |

## Local Development Workflow

Use this command when working only on the frontend:

```bash
npm run dev
```

Use this command when working with local persistence:

```bash
npm run dev:full
```

`npm run dev:full` starts:

- Vite frontend.
- JSON Server at `http://localhost:3001`.

The development API URL is loaded from `.env.development`:

```env
VITE_API_BASE_URL=http://localhost:3001
```

## Deployment Configuration

Production builds are generated with:

```bash
npm run build
```

Vite automatically loads `.env.production` during production builds:

```env
VITE_APP_ENV=production
VITE_API_BASE_URL=https://TU_BACKEND_PUBLICO.com
VITE_ENABLE_API_DEBUG=false
```

Before deploying to GitHub Pages, replace `https://TU_BACKEND_PUBLICO.com` with:

- A deployed REST API URL, or
- A public ngrok URL that exposes JSON Server, for example `https://example.ngrok-free.app`.

## GitHub Pages Compatibility

The router uses:

```js
createWebHashHistory(import.meta.env.BASE_URL)
```

The Vite base path is configured in `vite.config.js` from:

```env
VITE_APP_BASE_PATH=/biotrack-frontend/
```

This avoids broken routes when the app is hosted under a GitHub Pages repository path.

## Backend Configuration

### Local Mock Backend

JSON Server uses:

```bash
npm run server
```

The server reads:

```text
server/db.json
server/routes.json
```

Local frontend calls go to:

```text
http://localhost:3001
```

### Public Backend / ngrok

To expose JSON Server for a production demo:

```bash
npm run server
ngrok http 3001
```

Then update `.env.production`:

```env
VITE_API_BASE_URL=https://your-ngrok-url.ngrok-free.app
```

After changing `.env.production`, rebuild the frontend:

```bash
npm run build
```

## Centralized Runtime Configuration

`src/config/env.js` is the single source of truth for environment values:

- API base URL.
- API timeout.
- GitHub Pages base path.
- Application environment name.
- Debug flags.

`src/shared/infrastructure/api.service.js` is the single Axios gateway:

- Applies `baseURL`.
- Applies default headers.
- Applies timeout.
- Normalizes API errors.
- Enables optional debug logs.

Presentation components must not read environment variables directly and must not call Axios directly.

### Axios Usage Example

Infrastructure services should consume the centralized `apiService` with relative URLs:

```js
import { apiService } from '../../shared/infrastructure/api.service'

export const identityAccessApiService = {
  async fetchUserById(userId) {
    return apiService.get(`/users/${userId}`)
  },
}
```

The final request URL is resolved from:

```text
VITE_API_BASE_URL + /users/:id
```

For example:

```text
http://localhost:3001/users/1
https://your-ngrok-url.ngrok-free.app/users/1
```

If a future module uses `fetch`, it should still read the base URL from `src/config/env.js`:

```js
import { appEnvironment } from '../config/env'

const response = await fetch(`${appEnvironment.apiBaseUrl}/users`)
```

## Dependency and Version Management

Dependencies are declared in `package.json`. The project should be executed with Node.js 22 LTS because Vite 8 requires Node `20.19+` or `22.12+`.

Recommended local setup:

```bash
node -v
npm install
npm run dev:full
```

## Configuration Lifecycle

1. Developers use `.env.development` for local work.
2. Production builds use `.env.production`.
3. API URLs are changed only through environment files.
4. Shared HTTP behavior is changed only in `ApiService`.
5. Deployment base path is changed only through `VITE_APP_BASE_PATH`.
6. GitHub Pages deployment publishes the generated `dist/` directory.
